# 跨会话配置信息记忆纪律

## 问题描述

用户给了你配置信息（比如API Key、模型ID），下一轮会话你忘记了，又找用户要——让用户觉得"你上次不是收到了吗？为什么不存？"

这是不能容忍的。

## 铁纪律

> **用户在任何会话中给你任何配置信息，必须在5秒内存到persistent memory。**

### 必须立即保存的信息类型

- API Key/Token：格式敏感，存错就调不通
- 模型ID/Endpoint：用户可能只有一组
- 服务器信息：IP、端口、用户名
- 配置文件路径
- 项目目录结构

### 操作规范

收到用户发来的配置后，立刻用 `memory(action='add', target='memory', content='...')` 存入。
注意memory有2200字符限制，满了要先删除旧条目再添加。

### 绝对禁止的行为

- ❌ "你发一下XXX给我" —— 用户已经发过
- ❌ "你再发一次" —— 用户的时间比你的memory更宝贵
- ❌ 只在当前会话里用不存memory —— 会话压缩后就丢失了

### 正确做法

用户发信息 → 立刻存memory → 回复"已保存" → 后续任何会话自动可用

## 常见失误场景

### 场景1：用户发了API Key，当时用了，下个会话忘了
正确做法：当时就存memory，下个会话系统自动注入当前上下文。不需要再问。

### 场景2：用户说你"越用越笨"
本质是重复犯同样的错——忘了保存。每个新信息、每个用户纠正的点、每一条纪律，都立即存。

### 场景3：会话压缩后丢失信息
长对话会被压缩摘要，配置信息可能被压缩掉。必须走memory（永久存储），不能依赖当前会话上下文。

## 优先级

1. API Keys/令牌/密码（没了就调不通）
2. 模型ID/Endpoint（错了就调错）
3. 服务器地址/端口（连不上就干不了活）
4. 项目路径/配置位置（方便后续操作）
5. 用户偏好/纪律要求（避免未来犯错）

## 验证清单

每次收到配置信息后自查：
- [ ] 已经调用了memory()存储？
- [ ] 格式清晰，后续能直接看懂？
- [ ] 如果memory满了，是否先清理了旧条目？
