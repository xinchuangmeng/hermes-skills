# 反合理化 & 范围纪律检查

> 结合：Addy Osmani - Agent Skills (2026) + 生产环境Agent实践经验

## 范围纪律（Scope Discipline）

**黄金法则：只碰让你碰的东西。**

> "范围纪律是决定Agent的PR能否被merge的最重要因素。"

### 常见范围膨胀模式

| 问题 | 示例 | 后果 |
|------|------|------|
| 过度重构 | "这个函数写得不好，我顺便重构一下" | 审查范围爆炸，难以合并 |
| 顺手修改 | "看到这个TODO，我顺便实现一下" | 引入未经测试的代码 |
| 删除看不懂的代码 | "这段代码看起来没用"（其实是Chesterton's Fence） | 破坏隐藏的依赖 |
| 范围变更不通知 | "既然做A，顺便把B也修了" | 需求蔓延，PR reviewer需要重新理解上下文 |
| 优化预警 | "我知道我们只需要X，但我已经开始实现Y了——它很酷" | 时间浪费在不需要的功能上 |

### 修复后检查清单（改完代码后自查）

在执行任何代码变更后，回答以下问题：

1. ✅ 我改变了文件的数量是否等于需要改变的数量？（不多不少）
2. ✅ 我是否修改了任何不在任务范围内的文件？
3. ✅ 我是否删除了任何我不完全理解的代码？
4. ✅ 我是否添加了任务没有要求的功能？
5. ✅ 这个改动能用一句话解释清楚吗？（超过一句话说明范围有问题）

### 在Hermes中的实施

在`write_file`或`patch`之前，先问自己：
- 这个改动是不是任务描述里要求的？
- 如果不在任务描述里，我为什么要做这个改动？
- 用户/任务会说"谢谢你做这个"还是"为什么要做这个"？

如果答案是后者 → **停下来**。

## 反合理化表（Anti-Rationalization Tables）

### 写代码前的借口检查

| 借口 | 反驳 |
|------|------|
| "这个任务太简单了，不需要spec" | 验收标准仍然需要。写5行也比0行好。 |
| "我先写代码，测试后面补" | "后面"是最承重的词。没有后面。先写失败的测试。 |
| "测试通过了，可以发布了" | 测试通过是证据不是证明——检查runtime了吗？有人review代码了吗？ |
| "看起来没问题" | "看起来"从来不是充分条件。 |
| "这个功能很酷，虽然没要求但我加了" | 没有要求就是不要做。酷不酷不是你决定的。 |
| "我只是重构一下，功能不会变" | 重构是风险最大的变更之一——有没有测试覆盖？ |
| "这块代码太乱了，我重写吧" | 混乱但有注释的代码比整洁但没人懂的代码好。 |
| "改这个bug正好顺便把相关的也修了" | 一次只修一个bug。分开提交。分别测试。 |
| "时间不够了，这部分就不测了" | 不测的部分最有可能出bug。 |
| "这个错误很常见，我知道怎么修" | 常见的错误也可能有非常见的原因——先确认根因。 |

### Agent特别容易犯的借口

| Agent借口 | 反驳 |
|-----------|------|
| "根据最佳实践，我应该..." | 你的任务不是应用通用最佳实践，是完成指定的任务。 |
| "作为一个AI，我发现了一个相关问题..." | 发现了就记录在备注里，不要混进当前的改动中。 |
| "为了提高代码质量，我..." | 代码质量提高不是为了让你添加未请求的功能。 |
| "我可以顺便优化一下性能" | 如果性能不在任务要求中，那就不是你的工作。 |

## 使用方式

### 作为独立检查（推荐）
在执行任何代码变更前，先过一遍反合理化表和范围纪律检查清单。

### 作为技能内容嵌入
把你创建的每个技能（skill-biz-*）末尾加上反合理化表。

### 作为delegate_task的指令
```python
delegate_task(goal="修复X bug", context="范围纪律：只修改有bug的那一行。不要重构、不要优化、不要修改任何其他文件。完成前自查范围纪律清单。")
```

## 注意事项
⚠️ 反合理化表要针对性地写——通用表格效力有限
⚠️ 范围纪律不是"什么都别做"——如果你的任务明确要求重构，放心做
⚠️ 当时间充裕且测试覆盖完善时，可以适度放宽——但始终保持"一个改动一个PR"
⚠️ 这个技能最适合用于给subagent下指令时——显式告知范围边界
