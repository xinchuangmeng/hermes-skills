# Skill v2.0深度升级模式（离线版）

## 🎯 适用场景

1. **网络搜索不可用**（Tavily报432/429/403等错误，浏览器和web_extract也失败）
2. **技能文件较大**（600行+的大型SKILL.md），需要增量更新而非重写
3. **技能有编号层级**（如 `4.1, 4.2` 需要随着章节插入变为 `5.1, 5.2`），存在级联修复需求
4. **需要基于LLM内部知识补充内容**（而非网络搜索结果）

## ⚖️ 核心原则

| 场景 | 原则 |
|------|------|
| 搜索可用时 | 优先搜索：至少3个不同维度搜索，获取权威来源 |
| 搜索不可用时 | ❗不空转等待 → 先用LLM知识库做deep upgrade，标记数据源 |
| 内容来源标记 | 来源列表更新日期+注明"vX深度升级"，如搜索恢复可补来源链接 |

## 🔧 操作流程

### Phase 1：诊断（先读，再动手）

```yaml
1. skill_view(name) → 获取完整SKILL.md和文件结构
2. 评估现有内容的：
   - 时效性（数据/来源的日期是否过时）
   - 完整性（是否有明显遗漏的新趋势/新平台/新工具）
   - 结构性（章节编号是否连续、层级是否一致）
   - 实用性（是否有可操作表格和流程）
3. 确定需要更新的模块（不是重写，是增量补充）
```

### Phase 2：知识补充策略（离线版）

**当搜索不可用时，知识补充来源优先级：**

```
最高优先级：session_search（查过往对话中是否有相关讨论）
次高优先级：skill_view on related_skills（加载关联技能，交叉参考获得深度和互补视角）
             └─ 用现有技能schema中的related_skills字段找到关联技能
             └─ 同一领域下不同技能从不同角度切入，综合后产生新认知
第三优先级：LLM训练知识（当前模型知识截止日期后的内容标记为"推测/预估"）
最低优先级：跳过该模块（不如不做，不做错误的内容）
```

**实操建议：**
- 用 `session_search` 查所有相关过往session
- **用`skill_view`逐次加载所有related_skills**，注意这些技能可能从不同角度覆盖该领域——交叉阅读能发现重复内容（需去重）和互补视角（需整合）
- 对2026年时效性高的数据（GMV/月活），标注"预估"或"趋势判断"
- 避免编造具体数字——宁可写"持续增长中"也不要虚构数据点
- 在数据来源区增加一行说明：`- 部分2026年中数据基于行业趋势分析（搜索不可用时补充）`

### Phase 3：内容注入（patch策略）

**对大文件的patch原则：**

```
1. 找到明确的锚点字符串（必须是文件中唯一的文本）
2. 从「替换锚点」改为「在锚点后插入」= 用 old_string 包含锚点，new_string 包含锚点+新内容
3. 一次只改一个模块，改完立刻验证
4. 使用 read_file(offset, limit) 分段验证每个patch的结果
```

**常见patch模式：**

| 需求 | patch策略 | 示例 |
|------|-----------|------|
| 在章节前插入新章节 | old=现有章节标题，new=新章节+旧标题 | ✅ 最常用的模式 |
| 更新表格数据 | old=旧表行，new=新表行（含旧行） | ✅ |
| 更新单个字段（如description/tags） | old=旧字段值，new=新值 | ✅ 最简单 |
| 替换多处相同标签 | replace_all=true | ⚠️ 会造成误伤，谨慎使用 |

### Phase 4：编号一致性修复（最关键/最易错）

**编号错乱的常见原因：** 在新章节之间插入内容后，后面的章节编号需要整体后移。

**修复策略（不要一次性修全部）：**

```yaml
错误方法：写一个巨大的old_string覆盖所有编号（patch会失败，因匹配太长）
正确方法：
  每个章节标题单独patch，从前往后修
  # 示例：当在 section 4 前插入了 section 4（新的），旧 section 4→5
  patch 1: "## 四、" → "## 五、"  
  patch 2: "### 4.1" → "### 5.1"
  patch 3: "### 4.2" → "### 5.2"
  ...
```

**修复后验证：** 用 `read_file` 跳转检查：
- 每个 `## ` 标题是否连续（四→五→六...）
- 每个 `### X.Y` 的X是否匹配父级 `##` 的编号
- 所有内部引用（如"详见4.3节"）是否仍然正确

### Phase 5：YAML frontmatter更新

```yaml
# 必须同步更新：
description: 要在title之后包含"（v2.0深度升级版）"，并列出新增内容
tags: 添加新关键词（如新增平台的名称、新的核心概念）
title: 可选更新，如果原title太窄（如"四大平台"→"六大平台"），需要更新
```

### Phase 6：进度文件更新

更新 `auto_learn_progress.txt` 等进度文件时：
- 包含版本号标记（v2.0）
- 简述升级要点（新增板块/内容范围）
- 方便后续查阅此次升级的具体改动

## ⚠️ 避坑清单

| 坑 | 后果 | 预防 |
|----|------|------|
| patch old_string 在多处匹配 | 替换全部，产生乱码 | 先 read_file 确认唯一性；用 `grep` / `search_files` 检查重复 |
| 先改后修编号 | 改完后编号混乱，修复困难 | 插入前规划好编号位置，同批次完成 |
| 技能文件过大导致 read_file 慢 | 反复用 offset/limit 定位 | 先用 `wc -l` 看行数，用 `grep -n` 精确定位行号 |
| YAML 里有中文冒号/引号未转义 | skill_manage 解析失败 | em dash（——）不能在 YAML description 里出现；检查引号配对 |
| 忘记更新 tags | skill 搜索时找不到新内容 | 每次更新内容时必须同步更新 tags |
| 进度文件信息太简略 | 后续轮次不知道升级了什么 | 记录 \"新增X个板块/更新了Y数据/编号从Z升级到W\" |
| **⚠️ read_file行号前缀混入old_string** | 双前缀格式错乱 | `read_file`显示`   20|内容`但文件实际存`内容`不含前缀；复制old_string时必须去掉行号前缀。修复方法：先`sed -i 's/^    [0-9]*|    [0-9]*|/    |/' progress.txt`再Python重整行号 |

## ✅ 验证清单

- [ ] skill_view 可以正常加载，无YAML解析错误
- [ ] description 准确反映了v2.0升级内容
- [ ] 章节编号连续无跳跃（如 `## 四` 后面是 `## 五`）
- [ ] 所有数据源注明日期和版本
- [ ] 进度文件已更新
- [ ] 如果有影响到的关联技能（如引用此技能的模板），也一并检查
