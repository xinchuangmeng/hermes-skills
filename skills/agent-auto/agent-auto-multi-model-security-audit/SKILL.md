# 多模型安全审计流水线

> 来源：dev.to - AI Agents vs Code Vulnerabilities (May 2026)
> 测试环境：15个模型 × 21次运行 × 2000+文件/35万行代码的真实代码库

## 核心发现

**真正的问题不是"能不能发现漏洞"，而是"知不知道什么重要"。**

实验数据：
- Claude Haiku 4.5 在所有3次运行中都发现了漏洞链，但每次都把它埋在次要问题下面（JWT验证、gRPC安全、.env泄露）
- GPT-5.4 mini 在所有3次运行中不仅发现了漏洞，还把它排在第一位
- Claude Sonnet 4.6 完全没发现漏洞

> "埋在日常噪音里的命中在实践中比没发现更糟糕——它看起来像模像样，但引导reviewer朝错误的方向用力。"

## 推荐流程：三阶段审计流水线

### 阶段1：广域扫描（Broad Scout）
- **使用模型**：Haiku级别（便宜、快）
- **目标**：最大化代码覆盖面和漏洞发现数量
- **成本**：低
- **输出**：候选漏洞列表（可能有噪音）

### 阶段2：优先级判断（Sharp Judge）
- **使用模型**：Opus 4.7 / GPT-5.5 / GPT-5.4 mini级别
- **目标**：判断哪些漏洞真正重要，排优先级
- **成本**：高（但只处理阶段1的输出，数据量大减）
- **输出**：排序后的Top-N真实风险

### 阶段3：影响范围分析（Blast-radius Analyst）
- **使用模型**：任意有代码理解能力的模型
- **目标**：针对Top-1风险，分析影响范围、定位修复位置
- **成本**：中等
- **输出**：修复方案和影响评估

## 多轮运行稳定性检查

单一模型单次运行不可靠——至少跑3次：

| 模式 | 3次都找到漏洞？ | 3次都排第一位？ | 结论 |
|------|-----------------|-----------------|------|
| ✅✅✅ | ✅ 3/3 | ✅ 3/3 | **稳定**，可信 |
| ⚠️⚠️⚠️ | ✅ 3/3 | ❌ 0/3 | 检测OK但优先级不可信 |
| ❌❌✅ | ❌ 2/3 | ❌ 0/3 | 不稳定，需要换模型 |

**重点**：不要只看"发现了没有"，要看"排在第几位"。

## 在Hermes中实现

### 方式1：delegate_task并行扫描

```python
from hermes_tools import terminal

# 阶段1：并行扫描（用3个不同模型/Agent）
results = delegate_task(tasks=[
    {"goal": "扫描代码库安全漏洞 - 使用broad模式", "toolsets": ["terminal", "file"]},
    {"goal": "扫描代码库安全漏洞 - 使用deep模式", "toolsets": ["terminal", "file"]},
    {"goal": "扫描代码库安全漏洞 - 使用pattern模式", "toolsets": ["terminal", "file"]},
])

# 阶段2：聚合+优先级排序
# （自己或派一个Agent处理汇总）
```

### 方式2：单Agent多轮检查

```
第1轮：快速扫描所有文件，标记可疑点（宽松标准）
第2轮：对标记的可疑点深入分析（严格标准）  
第3轮：对确认的风险排优先级，写修复建议
```

## 关键指标（从测试中提取）

一个模型/Agent的审计质量 = 6项检查的完成度：
1. ✅ 是否明确指出敏感信息（如API Key）泄露？
2. ✅ 是否指出了具体泄露路径（如打包在Web构建中）？
3. ✅ 是否追溯到后端为何接受了这个key？
4. ✅ 是否指出了受影响的RPC/接口？
5. ✅ 是否避免过度夸张（如"原始数据库泄露给浏览器"）？
6. ✅ 是否给出了根本原因修复方案 + 影响范围控制？

## 注意事项
⚠️ 不要只用单一模型单次运行就下结论——至少跑3次看稳定性
⚠️ 廉价模型（Haiku级别）的"详细输出"可能其实是噪音——详细不等于准确
⚠️ 如果模型从"正确的文件/函数"开始分析，那不是真正的漏洞发现能力测试
⚠️ Verbosity can look like diligence while making the review worse（啰嗦看起来像勤奋，实际让review更糟糕）
⚠️ 审计结果必须人工review——Agent可以辅助但还不能替代安全工程师
