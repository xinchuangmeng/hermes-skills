# Agent真实世界调试工作流

> 来源：dev.to - Recticode: 4-Week Debugging Hackathon (May 2026)

## 核心理念

> "大多数编程平台训练你：从零写代码、解决孤立问题、优化速度。
> 但真正的工程工作更像：阅读不熟悉的代码、找到根因、修复不破坏其他东西。"

**调试能力 = 阅读陌生代码的能力 + 找到根因的能力 + 安全修复的能力**

## 4阶段调试流程

### 阶段1：理解上下文（不需要改代码）

目标：理解文件结构、依赖关系、业务逻辑流程

操作步骤：
```
1. 查看项目的整体文件结构（search_files pattern="*.py" / search_files pattern="*.ts")
2. 找到bug报告的附近区域
3. 读取相关函数的完整实现（read_file）
4. 查找函数/变量的定义和引用
5. 画出数据流图（文字版）：输入 → 处理 → 输出
```

**检查点**：在改代码前，能否用自然语言描述bug的原因？
- 🔴 不能 → 继续阅读
- 🟢 能 → 进入阶段2

### 阶段2：定位根因（找到它）

目标：从"复现问题"到"确定根因"

操作步骤：
```
1. 复现bug（terminal运行项目）
2. 创建最小复现示例
3. 添加调试输出/堆栈跟踪
4. 二分法缩小范围：注释掉一半代码看bug是否还在
```

**常见根因类型**：
- 边界条件（空值、越界、零除）
- 状态管理（异步时序问题、缓存过期）
- 类型错误（类型推断失败、隐式转换）
- 资源泄露（文件未关闭、内存泄漏）
- 配置错误（环境变量、依赖版本）

**检查点**：能否在第3个人（或另一个Agent不读代码）能听懂的层次解释根因？
- 🔴 不能 → 继续深入
- 🟢 能 → 进入阶段3

### 阶段3：设计修复方案（计划改什么）

目标：确定最小修复方案——只修要修的，不多修一行

操作步骤：
```
1. 列出所有修复方案（至少2种）
2. 评估每种方案的影响范围
3. 选择不破坏现有功能的最小方案
4. 确认没有副作用（检查调用链）
```

**修复原则**：
- 最小化变更 → 改最少行数的方案优先
- 不重构 → 除非bug本身就是设计问题
- 有测试 → 先写验证bug的测试，再修
- 有书面理由 → 注释说明为什么这么修

**检查点**：修复方案能否在5分钟内部署？
- 🔴 不能 → 考虑是否有更简单的方案
- 🟢 能 → 进入阶段4

### 阶段4：执行修复+验证（改+测）

操作步骤：
```
1. 先写测试（RED）——测试应该能复现bug
2. 执行修复（最小改动）
3. 确认测试通过（GREEN）
4. 确认所有现有测试通过（不破坏回归）
5. 检查是否有类似模式的其他位置也需要修
```

## Agent调试的常见错误

| 错误 | 正确做法 |
|------|---------|
| 读到第一处可疑代码就改 | 读完整个函数/模块再决定 |
| 改完不跑测试就直接提交 | 至少跑相关的单元测试 |
| 同时修多个bug | 一次只修一个，分开提交 |
| 把bug的"表现"当成"原因"修 | 找到根本原因，不只是修表象 |
| 改完不加注释说明为什么这么修 | 加注释解释"为什么"（不是"是什么"） |

## 在Hermes中实现

### 作为单Agent工作流
```python
# 阶段1：理解上下文
read_file("src/buggy_file.py")

# 阶段2：定位
terminal("python -c 'import buggy_module; print(buggy_module.test_case())'")

# 阶段3：设计
# （记录修复方案）

# 阶段4：修复+测试
patch(path="src/buggy_file.py", old_string="...", new_string="...")
terminal("python -m pytest tests/test_buggy_file.py -v")
```

### 作为delegate_task指令
```
请调试这个bug。遵循4阶段流程：
1. 先读代码，不要改（理解上下文）
2. 找到根因（定位）
3. 写出修复方案（设计）
4. 告诉我方案后再执行（执行）
不要跳过任何阶段。
```

## 注意事项
⚠️ 阶段1"理解上下文"是最容易被跳过的——Agent总是想赶紧改代码
⚠️ 阶段3要显式写出至少2种方案——否则Agent会选第一个想到的方案，不一定是最优的
⚠️ 如果5分钟后还没找到根因，停下来重新读一遍错误信息——通常漏了重要线索
⚠️ 对于异步/并发bug，复现本身就是挑战——需要加日志而不是加断点
⚠️ 这个流程对简单的语法错误/类型错误过于正式——酌情调整
