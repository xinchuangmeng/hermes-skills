# Agent认知中间件模式——给编排平台添加认知能力

## 🎯 核心洞察

### 认知层 vs 执行层

Agent编排平台（n8n/Hermes）擅长的是**执行层**——调度节点、传递数据、处理结果。但它们缺乏**认知层**——理解问题、做推理、规划步骤。

```yaml
# 问题
n8n/Hermes可以: 在A节点收到数据 → 传递给B节点 → 触发C操作
n8n/Hermes不能: 理解"为什么用户问这个"、"应该怎么规划步骤"、"这个结果合理吗"
```

### 解决方案：认知中间件

> 在编排引擎外面包一层LLM驱动的认知层，让Agent编排既有确定性（编排引擎）又有灵活性（LLM推理）。

## 📋 4种集成模式

### 模式1：前置决策（Pre-decision）
**适用场景：** 执行前需要理解用户意图、选择路径

```yaml
工作流:
  用户输入 → [认知中间件] → 执行引擎
  
  认知中间件做的事情:
    1. 理解用户意图分类
    2. 决定走哪条执行路径
    3. 提取关键参数
    4. 将结构化指令传给n8n

# 示例：客服系统
用户: "我想退掉上周买的那个包"
认知层:
  - 意图: "退货"
  - 实体: "包", "上周"
  - 情绪: "正常，无投诉倾向"
  - 推荐路径: "退货流程A（标准退货）"
  
n8n执行:
  - 查询订单流水
  - 生成退货单
  - 发送退货指南到用户邮箱
```

### 模式2：并行推理（Parallel Reasoning）
**适用场景：** 同一任务需要多角度分析

```yaml
工作流:
  输入 → [并行分发] → [认知引擎A] → [认知引擎B] → [综合器] → 输出
                       → [认知引擎C]
  
  每个认知引擎负责不同角度:
  - A: 事实核查（验证输入中的陈述是否属实）
  - B: 情感分析（判断用户情绪和需求紧迫度）
  - C: 技术分析（提取技术细节）
  
  综合器: 汇总三个引擎的输出，生成最终决策

# 示例：内容审核Agent
输入: 用户提交的产品描述
并行推理:
  - 合规检查 → "包含敏感词，标记需人工"
  - 质量评分 → "内容质量4.5/5"
  - 分类识别 → "电子产品类"
综合输出: "标记人工审核 + 自动分配到电子产品类目"
```

### 模式3：反馈循环（Feedback Loop）
**适用场景：** 执行过程中需要动态调整

```yaml
工作流:
  执行引擎执行 → 中间结果 → 认知中间件评估 → [满意] → 继续
                                    ↓
                                 [不满意] → 调整参数 → 重新执行

# 示例：内容生成Agent
n8n执行: 调用LLM生成产品描述
认知中间件检查:
  - "是否包含3个以上利益点？" → 只包含2个 → 不满足
  - "是否有明确的CTA？" → 没有 → 不满足
  - [不满足] → 自动调整prompt重新生成
  - 最多重试3次，都失败则标记人工处理
```

### 模式4：分层编排（Layered Orchestration）
**适用场景：** 复杂多步任务

```yaml
工作流:
  顶层: [认知编排器] — 理解任务、制定计划、分配子任务
               ↓
  中间层: [子任务队列] — 按依赖关系排列
               ↓
  底层: [n8n执行节点] — 执行每个子任务
               ↓
  反馈: 执行结果 → [认知编排器] 评估 → 继续/调整/终止

# 示例：市场调研Agent
认知编排器:
  计划: 
    1. 搜索竞争对手信息
    2. 分析价格对比
    3. 生成市场报告

  n8n执行:
    步骤1: 搜索节点（Search API）× 3个关键词
    步骤2: 数据处理节点（汇总结果）
    步骤3: LLM节点（生成报告）

  评估: 
    - "搜索结果是否足够？" → 不够则补充搜索
    - "对比分析是否精确？" → 有误则重新分析
    - "报告质量是否达标？" → 不达标则重新生成
```

## 🔧 在Hermes中的对照实践

```yaml
# Hermes本身就是一个认知+执行一体化的Agent系统
# 理解这4种模式有助于你用Hermes构建更复杂的Agent

hermes_equivalent_patterns:
  前置决策:
    - Hermes用delegate_task传参来决定subagent的行为
    - 类似：先分类 → 再分发到专业subagent
  
  并行推理:
    - Hermes用delegate_task并行执行多个任务
    - 类似：同时对多个API做查询 → 汇总结果
  
  反馈循环:
    - Hermes的Agent循环本身是反馈循环
    - 类似：执行→检查→再执行→直到满意
  
  分层编排:
    - Hermes用skill_manage做技能分层
    - 类似：顶层编排器→技能层→工具层
```

## ⚠️ 注意事项

1. **认知中间件不是必须的** — 简单工作流不需要认知层，直接n8n编排就够了
2. **延迟增加** — 每多一层认知中间件就多一次LLM调用，增加了响应时间
3. **成本叠加** — 认知层的LLM调用成本需要计入总成本
4. **认知层可能出错** — LLM的幻觉会传导到执行层，要有容错机制
5. **模式不是互斥的** — 可以组合使用（如前置决策+反馈循环）
6. **从最简单的模式开始** — 先试前置决策模式，复杂度逐步增加
