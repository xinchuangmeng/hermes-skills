# 跨模型提示词可移植性（Prompt Portability）

> 来源：dev.to - "OpenAI Codex is Free Right Now. That's the Trap." (May 2026)

## 核心洞察

免费（或低价）AI工具的商业模式：
1. 让开发者深度集成到日常工作流
2. 团队形成肌肉记忆
3. PR review假设AI辅助输出
4. Sprint速度依赖该工具
5. 定价上线时，切换成本已高到无法承受

> "使用它。但不要需要它。这是唯一能让你在账单到来时保持灵活的策略。"

## 提示词可移植性检查清单

### ✅ 要做

1. **使用通用格式**：Markdown + 自然语言，不依赖特定模型的标记语法
2. **显式声明格式要求**：不要说"返回JSON"而要说"返回一个包含name和age字段的JSON对象"
3. **任务说明与格式分离**：先描述任务内容，再说明输出格式
4. **使用标准Role标签**：system/user/assistant（几乎所有模型都支持）
5. **为最差模型优化**：提示词能在最弱的模型上跑通，在强模型上自然更好

### ❌ 不要做

1. **不要用厂商特定语法**：如OpenAI的function calling格式、Anthropic的XML tags特定写法
2. **不要依赖特定工具的快捷指令**：如Codex的"/correct this function"——换个工具就没了
3. **不要把生产代码中硬编码API调用方式**：抽象一层Adapter/Provider模式
4. **不要用只有特定模型才理解的指令**：如"用chain-of-thought"（CoT在不同模型效果不同）

## 提示词兼容性矩阵

| 特性 | 在GPT能用？ | 在Claude能用？ | 在DeepSeek能用？ | 在开源模型能用？ |
|------|------------|---------------|-----------------|-----------------|
| 自然语言指令 | ✅ | ✅ | ✅ | ✅ |
| System Prompt | ✅ | ✅ | ✅ | ✅ (部分) |
| JSON mode | ✅ | ✅ (via prompt) | ✅ | ❌ (部分支持) |
| Function Calling | ✅ | ✅ (Tool Use) | ✅ | ❌ (格式不统一) |
| XML Tags | ⚠️ 部分理解 | ✅ 原生支持 | ⚠️ | ❌ |
| Structured Outputs | ✅ 原生 | ✅ (via schema) | ❌ | ❌ |

## 可移植提示词模板

```markdown
你是一个[角色]。请执行以下任务：

## 任务说明
[清晰的任务描述，避免歧义]

## 输入
[输入数据]

## 输出格式
请返回一个JSON对象，包含以下字段：
- field1: 类型说明
- field2: 类型说明

## 约束
- 不要编造信息
- 如果不确定，说"不确定"
```

这个模板在GPT/Claude/DeepSeek/开源模型上都能工作。

## 工具选择决策框架

```
检查你的AI工具依赖：
1. 如果明天这个工具涨价5倍，你会怎么办？
2. 如果这个工具下线了，你的工作流还能继续吗？
3. 你的提示词换个模型还能用吗？

如果以上任何问题是"不会/不能"，说明你已经有锁定了。
```

## 在Hermes中的实践

1. **System Prompt**：使用标准格式，避免针对特定提供商的写法
2. **技能文件（SKILL.md）**：使用标准YAML frontmatter + Markdown，可在不同Agent间移植
3. **Provider配置**：通过config.yaml的provider字段切换，不改动技能内容
4. **避免硬编码**：不把特定模型名称写入提示词逻辑

## 注意事项
⚠️ 可移植性 ≠ 通用性——不同模型的能力上限不同，最弱模型决定基线
⚠️ 如果某个特定模型的特性真能带来10倍效率，绑定它也是合理的选择——有意识的选择 vs 无意识的默认
⚠️ 不要为了可移植性牺牲质量——先选最佳模型，再优化提示词使其可移植
⚠️ 不同模型对相同提示词的响应质量差异可能很大——需要在每个模型上测试
