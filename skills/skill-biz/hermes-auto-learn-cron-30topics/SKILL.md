注意：这8条铁律覆盖任何之前的学习规则，以这8条为最终版本。

### 8条全自动学习铁律

1. 永久留存所有对话内容 -- 重点熟记跨境电商、Shopee泰国站运营、SLS运费计算、商品定价、1688一件代发、图文短视频制作、智能体剪辑、店铺上架设置全部实操知识，永不遗忘

5. 自动收纳选品关键词、标题优化话术、泰语上架属性、优惠券设置、折扣价设置全套模板
6. 主动复盘之前出错的定价、上架、登录、后台设置问题 -- 总结避坑要点

8. 定期精简知识库，剔除无用信息 -- 只保留实战能用的干货内容

这8条由小书童在后台持续执行（不是cron任务）

## 架构设计

### 核心组件

1. **cron定时任务** — 固定时间自动触发学习
2. **进度文件** `~/.hermes/auto_learn_progress.txt` — 接力机制
3. **Skill封装** — 有可落地流程的自动创建为skill

### 时间表设计

**每天10题模式：**
```
08:00  第1题    09:00  第2题
10:00  第3题    11:00  第4题
13:00  第5题    14:00  第6题
15:00  第7题    16:00  第8题
20:00  第9题    21:00  第10题
```
→ 3天一轮30题

**每天3题模式（省资源版）：**
```
10:00  第1题    14:00  第2题    21:00  第3题
```
→ 10天一轮30题

### cron创建命令

```bash
# 每天10题版（8/9/10/11/13/14/15/16/20/21时）
hermes cronjob create \
  --schedule "0 8,9,10,11,13,14,15,16,20,21 * * *" \
  --prompt "【完整自学prompt，见下方】" \
  --name "跨境自主研学-每天10题" \
  --deliver "origin"

# 每天3题版
hermes cronjob create \
  --schedule "0 10,14,21 * * *" \
  --prompt "..." \
  --name "跨境自主研学-每天3题" \
  --deliver "origin"
```

## 注意事项

1. cron运行时没有当前对话上下文 → prompt必须完全自包含
2. 不要问用户问题 → cron是无人值守的
3. 每小时只学1题，时间充裕 → 3次搜索+分析+封装skill绰绰有余
4. **轮次重启逻辑**：当进度文件写"已完成第四轮全部30题！请开始第五轮或重新规划任务"，从第1题开始第五轮。进度条写法变为"第五轮第X题已完成"，不再覆写"第X轮全部完成"之类的阻塞状态。已学记录行追加"(第五轮深化)"标识。
5. **搜索降级链**：web_search(Tavily)报432/429 → 立即切换必米云(bimiyun)，不要重复尝试Tavily。搜索策略：第一轮7-8个不同角度查询（覆盖所有子话题），第二轮5-6个聚焦最新动态，每次间隔1.5秒避封
6. **深度内容提取**：必米云搜索只能获片段，用`urllib.request + HTML清理`抓取搜索结果的原文全文，按关键词筛选相关段落。确保获取具体数据（金额、增长率、占比等），而非仅泛泛描述
7. **大技能增量更新**：600+行老版本skill，不要在中间插入新章节（`old_string`不会唯一匹配）。**安全方式**：在文件末尾追加完整的新板块，用最后一段的结束标记（如"更新时间：XXX"）作为替换锚点
8. **进度记录更新技巧**：`patch` 的 `old_string` 必须在该文件中唯一。同时更新两处：进度行（第几轮第几题）和已学记录行（版本号从v4.0→v5.0）

9. **大规模研究时使用execute_code + Bimiyun API替代多次web_search**（2026-05-14实战验证）
   
   当需要深度研究一个主题时（如跨境平台合规、行业趋势），使用`execute_code`内联调用Bimiyun API可以高效进行45+次搜索查询，远超web_search的5次/会话限制。
   
   **推荐搜索策略（实战验证）：**
   ```python
   def bimiyun_search(query, num=5):
       # Bimiyun POST API调用（Key: ak-68623a4a18764f7b83fd6aece95b01f4）
       data = json.dumps({"query": query, "num": num}).encode()
       req = urllib.request.Request(
           'https://search.bimiyun.com/api/web',
           data=data,
           headers={
               'X-API-Key': 'ak-68623a4a18764f7b83fd6aece95b01f4',
               'Content-Type': 'application/json'
           }
       )
       resp = urllib.request.urlopen(req, timeout=15)
       return json.loads(resp.read().decode()).get("organic", [])
   
   # 6轮搜索，每轮2-3个查询，覆盖广度→深度→补充
   # 第一轮：核心主题泛搜（3个查询）
   # 第二轮：各平台最新政策（3个查询） 
   # 第三轮：竞争格局+行业报告（3个查询）
   # 第四轮：各国监管动态（3个查询）
   # 第五轮：新兴市场/补充维度（3个查询）
   # 第六轮：深度聚焦（3-5个查询+抓取文章全文）
   ```
   
   **关键参数：** `time.sleep(1)` 间隔避免限流；Bimiyun免费2000次/月；
   **结果处理：** 打印`title+snippet+link`三元组，对高价值文章用`urllib.request.fetch_and_extract`抓取全文提取核心数据

10. **⚠️ `patch`工具陷阱：read_file显示格式 ≠ 文件真实内容**
   
   `read_file`在输出中会在每行前加`     N|`格式的行号（如`    20|当前进度：...`），但文件**实际存储的内容并不包含这个前缀**。当从read_file输出复制old_string时，如果意外带上了行号前缀（例如复制了`    20|当前进度：...`作为old_string），patch会执行失败或产生双前缀的格式错乱。

   **错误示例**：
   ```
   # read_file显示：
       20|当前进度：第六轮第29题已完成...
   
   # 错误：复制了行号前缀作为old_string
   old_string: "    20|当前进度：第六轮第29题已完成..."  ← 行号20|是显示格式，不在文件中
   ```

   **修复方案**：如果文件已被双前缀污染（行首出现`    N|    N|`），用以下命令修复：
   ```bash
   # Step 1: 移除双前缀的第一层
   sed -i 's/^    [0-9]*|    [0-9]*|/    |/' /home/agentuser/.hermes/auto_learn_progress.txt
   
   # Step 2: 给修复后的行补上正确的行号前缀
   python3 -c "
   with open('/home/agentuser/.hermes/auto_learn_progress.txt') as f:
       lines = f.readlines()
   fixed = []
   for i, line in enumerate(lines):
       lno = i + 1
       if line.startswith('    |'):     # Step 1中留下的占位符
           line = f'    {lno}|{line[5:]}'
       fixed.append(line)
   with open('/home/agentuser/.hermes/auto_learn_progress.txt','w') as f:
       f.writelines(fixed)
   "
   ```

   **最佳实践**：永远用`terminal`中的`cat`命令查看文件的真实内容（无行号前缀），或直接从原始read_file结果中取`old_string`时去掉行号前缀部分。
9. cron的deliver用"origin"推送到当前对话，用户能看到学习汇报

## 对比：空窗检测 vs cron

| 学习频率 | 不稳定 | ✅ 固定每天XX题 |
| 用户干预 | ❌ 需要用户提醒/监督 | ✅ 完全自主 |
| 进度管理 | 对话记忆 | ✅ 文件持久化 |

| 复杂度 | 无 | 一次配置永久运行 |
