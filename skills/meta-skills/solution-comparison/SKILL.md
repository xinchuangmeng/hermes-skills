# 方案对比元技能

## 原则
**每个问题至少给2种方案。** 不要只输出一种做法让用户没得选——列出多个方案的对比，包括推荐理由。

## 触发条件
- 技术选型（用什么工具、用什么库、用什么架构）
- 流程设计（先做A还是先做B）
- 资源分配策略
- 用户问"怎么做更好"或"有什么方案"

## 输出格式

```
## 可行的方案

### 方案A：[名称 + 一句话描述]
### 方案B：[名称 + 一句话描述]
### 方案C（可选）：[名称 + 一句话描述]

## 对比矩阵

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 实现成本 | | | |
| 可维护性 | | | |
| 适用规模 | | | |
| 边界条件/坑 | | | |
| 失败风险 | | | |

## 推荐方案
- 推荐：[方案名] 
- 理由：（简短说清楚为什么）
- 如果将来场景变了，什么时候应该切到其他方案
```

## 对比维度参考（根据场景选3-5个）

| 维度 | 适用场景 |
|------|---------|
| 实现成本（人时） | 所有 |
| 运行性能 | 选型+数据量大的场景 |
| 可维护性 | 长期维护项目 |
| 边界条件 | 所有（必填） |
| 失败风险/回退方案 | 涉及生产环境 |
| 扩展性 | 可能增长的业务 |
| 学习曲线 | 团队协同场景 |
| 依赖风险 | 引入第三方依赖 |

## 例子

❌ "用 Python 的 schedule 库做定时任务就行"

✅ 
```
方案A：Python schedule 库
方案B：Linux crontab + shell 脚本
方案C：systemd timer

方案A简单但进程不能重启，方案B稳定但日志要自己处理，方案C最正规但配置稍多。
推荐方案B——服务器上最稳定，你还会用。
```
