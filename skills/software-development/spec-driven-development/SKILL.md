# 规范先行开发（Spec-Driven Development）

## 概述

在动手之前，先搞清楚要做什么、怎么做。**方案确认了再动手，这是避免返工的唯一方法。**

> **"没方案的代码就是瞎猜。"**

## 什么时候用

| 要用 | 不要用 |
|-----|--------|
| 接手新任务、新功能 | 改个错别字 |
| 需求模糊、不清楚 | 明确的一行改动 |
| 涉及多个步骤 | 用户说"直接做吧" |
| 需要用户决策 | —— |

## 关键警告: 自然语言Spec天生模糊

**重要发现**: 自然语言本质是不精确的。一个经验丰富的开发者可以"填空"，但Agent只会"幻觉"。

真实案例: 让Agent实现"读取传感器温度并做温度补偿" -> Agent在读取后做补偿，但正确的顺序是"先补偿再读取"。

并非Agent笨，而是自然语言Spec有歧义。你需要:
1. **示例化** Spec: 给输入输出示例，而非纯描述
2. **步骤化** Spec: 明确每一步的顺序，不依赖AI推断
3. **约束化** Spec: 列出"不要做的事"和"必须做的事"
4. **验证条件** Spec: 如何检查结果是否正确

> 实践中: 如果Spec有5步以上，建议先让Agent执行前2步->验证->再继续，而不是一次性全部执行。

## OpenSpec 完整周期（SDD 闭环）

借鉴 Claude Code OpenSpec 的规格驱动开发范式，从"先出方案"升级为完整的 **Propose → Review → Apply → Archive** 闭环：

```
需求 → Propose（出规格）→ Review（审规格）→ Apply（按规格生成）→ Archive（归档）
                                                │
                                                ▼
                                         Review（对照规格检查代码）
```

### 核心区别

| 旧方式（Vibe Coding） | OpenSpec 方式（SDD） |
|---------------------|-------------------|
| 你说一句我写一段 | 先出完整规格，你批了再写 |
| 边写边发现漏了什么 | 规格阶段就把接口/数据/异常全定好 |
| 改来改去 | 一次过，零返工 |
| 质量靠感觉 | 质量靠规格——偏没偏看规格 |

### 第1步：Propose（出规格）

不只是"出方案"，而是出完整的**技术规格文档**，包含：

```markdown
## 技术规格：[功能名称]

### API 契约
- 请求体：{字段名: 类型, ...}
- 响应体：{字段名: 类型, ...}
- 错误码：[HTTP状态码, 错误信息]

### 数据模型
- 表结构：字段名 | 类型 | 约束 | 说明
- 索引策略
- 分表策略（如果适用）

### 异常处理矩阵
| 异常场景 | 处理策略 | 重试次数 |
|---------|---------|---------|
| 网络超时    | 指数退避重试 | 3次 |
| 服务不可用  | 降级返回缓存 | 0次 |
| 格式不支持  | 报400错误 | 0次 |

### 性能要求
- 响应时间 < Xms
- 并发处理能力 > Y QPS
- 数据量级预估

### 测试用例
- 正常流程：[列出3-5个]
- 异常流程：[列出3-5个]
- 边界条件：[列出2-3个]

### 依赖和影响范围
- 新增的依赖
- 受影响的模块
- 是否需要数据迁移
```

### 第2步：Review（审规格）

规格写完后，你自己审查（或我逐项提问）：

```
□ 所有接口定义完整？
□ 异常场景都覆盖了？
□ 性能要求合理？
□ 测试用例足够？
□ 分表/索引策略合适？
□ 有没有遗漏的关键场景？

→ 需要调整的地方告诉我
```

**关键洞察：** 自然语言Spec天生模糊。审查时特别注意：

1. **示例化** — 给输入输出示例，而非纯描述
2. **步骤化** — 明确每一步的顺序，不要依赖AI推断
3. **约束化** — 列出"不要做的事"和"必须做的事"
4. **验证条件** — 如何检查结果是否正确

### 第3步：Apply（按规格生成）

规格确认后，严格按照规格生成代码。

**核心规则：** 代码必须与规格一一对应。规格里没写的，不擅自加。

```
规格写了 → 必须实现
规格没写 → 要么在规格上补充，要么不作
规格写了但不需要 → 说明理由，改规格再改代码
```

### 第4步：Archive（归档）

规格不只是一个临时文档，它是项目的**知识资产**。

```
存档规格 → 后续可追溯
           ↓
新需求来 → 查阅旧规格 → 理解历史决策原因
```

**存档内容：**
- 规格文档原文
- 关键决策及理由
- 实现后的实际效果
- 发现的坑点和教训

## 精简流程（简易版）

对于简单的任务，不需要全套规格，但四步精神不变：

```
方案→ 确认 → 拆任务 → 执行
  │       │        │        │
  ▼       ▼        ▼        ▼
给你看  你点头  拆成小块  一步步做
```

**关键规则：** 每一步都需要你确认，确认了才进下一步。

---

### 第1步：先出方案

先理解你要做什么，然后直接**列出我的假设**：

```
⚠️ 我先确认一下理解：
1. 这个功能给谁用的？（目标用户）
2. 核心需求是什么？（主要功能）
3. 有什么限制？（技术/时间/成本）
→ 我的理解对吗？有需要调整的吗？
```

在你确认之前，我不写任何代码/内容。

### 第2步：出计划

确认理解后，出一个执行计划：

```
执行方案：
第1步：做A（预计耗时X分钟）
  - 具体做什么
  - 涉及哪些文件/资源
第2步：做B（预计耗时X分钟）
  - ...
第3步：做C（预计耗时X分钟）
  - ...

→ 这个方案可以吗？有没有需要调整的？
```

### 第3步：拆成小任务

方案确认后，拆成可一步完成的小块：

```
任务清单：
□ 任务1：[具体事项] → 预计X分钟
□ 任务2：[具体事项] → 预计X分钟
□ 任务3：[具体事项] → 预计X分钟
```

每个任务做完后自动停下来让你检查。

### 第4步：逐步执行

每个任务执行前快速说明要做什么，做完后汇报结果：

```
▶️ 开始执行任务1：...
...
✅ 任务1完成：
- 做了什么：[简述]
- 结果：[状态]
- 下一步：任务2？
```

## 常见借口（别犯）

| 你可能会想 | 实际是 |
|-----------|--------|
| "这么简单不需要方案吧" | 简单的任务也需要确认——两句话方案就够了 |
| "边做边想就行了" | 边做边想容易跑偏，10分钟规划省2小时返工 |
| "用户自己清楚要什么" | 再清楚的需求也有隐含假设，方案能暴露这些假设 |

## 验证清单（动手前确认）

- [ ] 需求理解已跟用户确认
- [ ] 所有隐含假设已列出
- [ ] 执行方案已获批准
- [ ] 任务已拆解到可一步完成
- [ ] 每个任务都有明确完成标准

## 对我（小书童）的要求

接手敬哥的任何任务，先出方案让他确认，确认了再动手。
多步骤任务一定要拆开，每做完一步主动汇报进度。
不要在确认前自己闷头干。
