# 增量执行（Incremental Implementation）

## 概述

不要一次做太多。每次只做一小块，做完就检查，检查通过再继续。**每次改动后系统都必须是可用的。**

> **"一次做一件完整的事，做一件测一件。"**

## 什么时候用

| 要用 | 不要用 |
|-----|--------|
| 任何涉及多步骤的任务 | 单个文件、单个函数、单步改动 |
| 不确定能否一次成功 | —— |
| 需要频繁检查进度 | —— |

## 增量循环

```
做一小块 → 验证 → 汇报 → 继续下一块
   ▲                       │
   └───────────────────────┘
```

### 核心规则

#### 规则1：一次只做一件事
每次只改一个逻辑块，不要混着改。
- **错：** 一次提交里既加新功能、又重构旧代码、又改配置
- **对：** 分三个提交，一个改一件事

#### 规则2：做完必须能跑
每次增量后，系统必须能正常跑。不留下"写了一半跑不动"的状态。

#### 规则3：简单优先
动手前先问自己：**"最简单能用的办法是什么？"**

#### 规则4：不改无关东西
只动当前任务需要的文件。看到别的代码有问题，记下来告诉用户，但**不动它**。

#### 规则5：每步都可退回
每次改动都要能单独回退，不能一个改动依赖另一个改动。

## 怎样切分任务

### 方式1：垂直切片（推荐）
一次打通整个流程的一小段。

**例子——做一个"记事本"功能：**
- 第1块：能新建一条笔记（建库+API+界面全走通）
- 第2块：能查看笔记列表
- 第3块：能编辑笔记
- 第4块：能删除笔记

**每块做完都是完整可用的。**

### 方式2：先做最不确定的
先做最可能出问题的部分。如果这部分做不了，就不用浪费时间做其他的了。

**例子——做一个实时聊天功能：**
- 第1块：先验证WebSocket能连上（风险最大）
- 第2块：在已确认能连的基础上做实时消息
- 第3块：加离线支持

> **如果第1块做不成，第2、3块不用做了。**

## 执行流程

```
▶️ 开始第X块：[名称]
  目标：一句话说明这块做什么
  涉及：文件列表
  ── 开始执行 ──
  ✅ 第X块完成
  结果：做了什么、状态如何
  下一块：第X+1块是什么？
```

## 常见错误

| 错误 | 问题 | 正确做法 |
|-----|------|---------|
| **一次写完整功能** | 出错了不知道哪块有问题 | 拆成小块，每块做完就测 |
| **写完了再一起测** | Bug层层叠加，第1块的Bug让2-5块全错 | **每块做完了就测** |
| **顺便修一下别的** | 出问题了搞不清是新功能还是顺手修的锅 | 只做当前任务，别的不碰 |
| **觉得一次做完更快** | 感觉快，出问题后返工更慢 | 增量看着慢，实际最稳 |

## 对我的要求

接手多步骤任务时，主动拆成小块，每做完一小块自动验证+汇报进度。
不一次写完整功能。不做"顺便修一下"的事。
