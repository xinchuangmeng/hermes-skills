# 结构化思考（Structured Thinking）

## 概述

**在动手之前，先想清楚要做什么、有几个选项、每个选项的代价是什么。**

这是从 Superpowers 的 `brainstorming` 元技能提炼的 Hermes 版本。核心原则：
**不要接受第一个冒出来的方案。先列举选项，再对比，再决策，最后动手。**

> **"匆忙动手的唯一回报是更匆忙地返工。"**

## 什么时候用

| 要用 | 不要用 |
|-----|--------|
| 需求模糊、要出方案 | 用户已经说清楚要什么 |
| 多个方案可选 | 只有一条路 |
| 边界条件和异常场景不清晰 | 简单的一步操作 |
| 用户问"有什么想法" | 用户问"怎么做"（方案已定） |
| 技术选型 / 工具选择 | 已知的最佳实践 |

## 核心流程：四步思考法

```
发散 → 收敛 → 对比 → 决策
  │      │      │      │
  ▼      ▼      ▼      ▼
列选项  筛重点  比优劣  定方案
```

---

### 第1步：发散——不要只给一个方案

面对任何开放式问题，先列出所有可行的选项。

**模板：**

```
## 可行方案

### 方案A：[名称]
核心思路：[一句话]
优点：[2-3点]
缺点/风险：[2-3点]
适用场景：[什么情况下这个方案最好]

### 方案B：[名称]
...

### 方案C：[名称]
...
```

**关键规则：**
- 至少给出 **2-3 个** 有实质差异的方案
- 不要为了凑数而编造弱方案
- 如果某个方案明显差，说明为什么差，但不删——用户可能有你不知道的偏好

**实际例子——"怎么提高跨境店铺转化率"：**

```
方案A：优化详情页
- 核心：主图A+测试、详情页文案重写、评价引导
- 优点：见效快（1-2周），成本低
- 缺点：有上限，不能质变
- 适用：转化率在1-3%之间

方案B：站内广告投放
- 核心：关键词竞价、关联广告、闪购活动
- 优点：流量精准，可量化ROI
- 缺点：烧钱，新手容易亏
- 适用：已有基础销量，想放大

方案C：内容引流（短视频+达人）
- 核心：TikTok内容创作、达人带货
- 优点：低成本获客，长期复利
- 缺点：见效慢（1-3月），需要内容能力
- 适用：Shopee/TikTok双平台，可互相导流
```

---

### 第2步：收敛——筛掉明显不行的

在用户反馈或已知约束基础上，快速排除：

```
❌ 排除方案C（用户没有内容团队，短期无法落地）
✓ 保留方案A和B
```

**排除标准：**
- 违反已知约束（技术/成本/时间/用户偏好）
- 需要不具备的能力/资源
- 风险不可控

**不要排除：**
- 用户没有明确表态的方案（让用户自己选）
- 你自己不喜欢的方案（保持中立）
- 看起来麻烦但可能有长远收益的方案

---

### 第3步：对比——做结构化对比

用表格做多维对比，让差异一目了然：

#### 对比矩阵模板

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| **实现难度** | ⭐⭐⭐ 中 | ⭐⭐ 低 | ⭐⭐⭐⭐ 高 |
| **见效速度** | 1-2周 | 立即 | 1-3月 |
| **成本** | 低（人力） | 中（广告费） | 低（内容） |
| **风险** | 低 | 中（可能亏） | 低 |
| **可扩展性** | 中 | 高 | 高 |
| **长期价值** | 中 | 低（停投即停） | 高（内容资产） |
| **与现有资源匹配度** | ✅ | ✅ | ❌（无团队） |

#### 场景对比

问自己：**在不同的情况下，哪个方案更优？**

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 预算充足 + 想快速起量 | B | 广告直接买量 |
| 预算有限 + 长期经营 | A | 先优化内功再考虑引流 |
| 有内容团队 | A+C | 两手抓 |

---

### 第4步：决策——给推荐方案 + 理由

```
## 推荐方案：A（优化详情页）

理由：
1. 用户还没有稳定的销量基础，直接投广告转化率低
2. 优化详情页是0风险投入，就算效果一般也不会亏钱
3. 先把内功练好，后续投广告的ROI才会高

建议执行顺序：
第1周：主图A+测试（至少3套方案）
第2周：详情页文案优化
第3周：评价引导策略
→ 转化率从X%提升到Y%后，再考虑投广告

→ 你觉得这个方向可以吗？还是想从别的角度切入？
```

**最后一定要问用户确认。** 结构化思考的目的是帮你决策，不是替你做决定。

## 三种常见思考框架

### 框架1：优劣势对比（Pros/Cons）

适用于：两个方案之间二选一

```
方案A vs 方案B

✅ 方案A的优势：
- ...

❌ 方案A的劣势：
- ...

✅ 方案B的优势：
- ...

❌ 方案B的劣势：
- ...
```

### 框架2：第一性原理（First Principles）

适用于：现有方案都有问题，需要从零思考

```
1. 核心目标是什么？（不能妥协的那个）
2. 完全从零开始，不考虑现有方案，最佳做法是什么？
3. 现有约束是什么？（必须遵守的）
4. 在约束内，离第2步的最佳做法最近的是什么？
```

### 框架3：边界条件扫描（Edge Case Analysis）

适用于：方案似乎可行，但不确定有没有隐藏问题

```
## 正常情况
- 用户按预期操作 → 走通

## 边界情况
- 输入为空 → ？
- 输入超长 → ？
- 并发操作 → ？
- 网络中断 → ？
- 权限不足 → ？
- 数据不存在 → ？
- 第三方服务超时 → ？
- 磁盘写满 → ？

## 每个边界情况
- 当前方案能处理吗？
- 不能的话需要加什么处理？
```

## 快速参考卡

### 四步口诀

```
先发散 再收敛
列方案 做对比
给推荐 问你意

不盲动 不武断
想清楚 再动手
比返工 快三倍
```

### 反模式清单

| 错误做法 | 为什么错 | 正确做法 |
|---------|---------|---------|
| 只给一个方案 | 用户没有选择权，错了全错 | 至少给2-3个有差异的方案 |
| 给太多方案（5+） | 选择困难，信息过载 | 发散时多列，呈现时只保留有意义的 |
| 直接说"方案A最好" | 跳过对比过程，用户看不清为什么 | 用对比矩阵展示差异 |
| 忽略边界条件 | 看似可行的方案，上线就崩 | 特意做边界条件扫描 |
| 推荐方案不说明理由 | 用户不知道怎么判断 | "推荐A因为...，不推荐B因为..." |
| 给完方案不等回复就动手 | 错误的方案执行得再好也是错的 | 必须等用户确认再动手 |
