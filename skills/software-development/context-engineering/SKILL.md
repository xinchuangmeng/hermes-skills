# 上下文工程（Context Engineering）

## 概述

> **"上下文是影响AI输出质量的单一最大杠杆——太少AI会瞎猜，太多AI会迷失。"**

上下文工程就是有意识地组织给AI的信息：什么时候给什么信息、怎么组织。

## 核心思想：信息分5层

从最持久到最临时，层层递进：

```
第1层：项目规则/用户偏好     ← 一直加载，项目级
第2层：需求文档/方案          ← 每次新功能加载
第3层：相关源文件             ← 每个具体任务加载
第4层：报错信息/测试结果      ← 每次迭代加载
第5层：对话历史               ← 不断积累，需压缩
```

### 第1层：规则和偏好（最高杠杆）
持久存放的信息，每次对话都自动加载：
- 用户偏好（语言、沟通风格）
- 项目目录结构
- 常用命令
- 必须遵守的规则

**我的做法：** 存在memory里，每次对话自动有。

### 第2层：需求文档
- 只加载当前任务相关的部分，不要一次性扔出整份文档
- **好的做法：** "以下是需求文档中关于用户注册的部分：..."
- **坏的做法：** 扔出5000字完整需求文档，让AI自己找

### 第3层：相关源文件
- 修改文件前先读一遍
- 实现新模式前先找找已有的类似例子
- 看懂了再动手

### 第4层：报错信息
- **好的做法：** 给具体的报错：`TypeError: Cannot read property 'id' of undefined at UserService.ts:42`
- **坏的做法：** 扔出500行测试输出，里面只有一行报错

### 第5层：对话管理
- 话题跑偏时主动总结压缩
- 新任务开新对话
- 上下文太大时主动归档重置

## 三种上下文组织策略

### 策略1：会话开场脑暴（新任务开始）

当开始一个新任务时，用这个模板组织信息：

```
项目背景：
- 我们在做 [项目名]，用的是 [技术栈]
- 相关的需求文档是：[摘录关键部分]
- 关键约束：[列出来]
- 涉及的文件：[文件名+简要说明]
- 已知坑点：[要注意的事情]
```

### 策略2：按需精选（每个具体任务）

```
任务：做 [具体事情]
相关文件：
- src/file1.py（要修改的目标文件）
- src/lib/utils.py（已有的工具函数）
- tests/test_file1.py（已有的测试）
参考模式：
- 看 src/lib/validation.py:45-60 是怎么做验证的
约束：
- 必须用已有的 ValidationError 类，不要抛新错误类型
```

### 策略3：分层摘要（大型项目）

维护一个摘要索引，每个任务只加载相关的部分。

## 信息冲突时怎么办

**不要默默选一个。要主动提出：**

```
⚠️ 信息冲突：
需求文档说要REST API，但现有代码用的是GraphQL（src/graphql/user.ts）。
选项：
A) 按需求文档走——加REST端点，以后可能废弃GraphQL
B) 按现有模式走——用GraphQL，更新需求文档
C) 先问你——这个决定不该我替你做
→ 选哪个方案？
```

## 需求不完整时怎么办

```
⚠️ 缺少需求：
需求文档定义了创建任务，但没说明重复标题怎么处理。
选项：
A) 允许重复（最简单）
B) 拒绝并报错（最严格）
C) 自动加编号后缀"任务(2)"（最用户友好）
→ 你想要哪种？
```

## 执行前出小方案

对于多步骤任务，先出个轻量级计划再执行：

```
执行方案：
1. 先做A——创建数据库表结构
2. 再做B——写API接口
3. 最后做C——前端对接
→ 这样行不行？需要调整吗？
```

> **"花30秒确认方案，省得花30分钟返工。"**

## 常见错误

| 错误做法 | 问题 | 正确做法 |
|---------|------|---------|
| **信息太少** | AI会自己编造API、忽略约定 | 每次任务前加载规则+相关文件 |
| **信息过量** | AI读5000行无关代码，迷失重点 | 只喂当前任务相关的部分 |
| **一次性扔所有需求** | AI记不住、抓不住重点 | 分层给，先给大方向再给细节 |
| **不交代背景直接派活** | AI误会你的意图 | 用"项目背景：..."模板开场 |
| **冲突信息不处理** | AI会随机选一个，往往选错 | 主动标出冲突让你决策 |

## 对我的要求

给用户派活时，信息要按层次组织，先给核心再给细节。
遇到模糊的需求或冲突的信息，先问清楚再动手，不要擅自做决定。
多步骤任务先出方案让用户确认，确认了再执行。
